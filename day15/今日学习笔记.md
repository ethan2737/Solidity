## 1. 什么是ERC20代币

### 1.1 概念理解

**ERC20 = 一套"代币标准"**

ERC20 是 Ethereum Request for Comments 第20号提案，定义了以太坊上代币的技术标准。它规定了代币必须有哪些功能，让所有代币"互相兼容"。

#### 简单比喻

| 现实世界 | ERC20 代币世界 |
|---------|---------------|
| 以太坊 | 去中心化的"银行系统" |
| ERC20 代币 | 银行发行的"存折" |
| ERC20 标准 | 存折的"使用规则" |

#### 为什么需要标准？

```
没有标准之前：
❌ 比特币接口 send()，以太坊接口 transfer()，莱特币接口 sendMoney()
    每种代币接口不同，钱包和交易所需要单独适配

有了 ERC20 之后：
✅ 所有代币都用：transfer(to, amount)
✅ 钱包只需写一次代码，就能支持所有 ERC20 代币
✅ 交易所可以快速上线新代币
✅ DeFi 协议可以与任何 ERC20 代币交互
```

### 1.2 ERC20 标准接口

ERC20 定义了**6 个函数 + 2 个事件**：

#### 必须的函数

| 函数 | 作用 |
|------|------|
| `totalSupply` | 代币总供应量 |
| `balanceOf(地址)` | 查询某个地址的余额 |
| `transfer(地址, 数量)` | 转账 |
| `approve(地址, 数量)` | 授权 |
| `transferFrom(从, 到, 数量)` | 被授权后转账 |
| `allowance(所有者, 使用者)` | 查询授权额度 |

#### 必须的事件

| 事件 | 触发时机 |
|------|----------|
| `Transfer` | 转账时触发 |
| `Approval` | 授权时触发 |

### 1.3 核心机制：approve + transferFrom

```
用户 A                    合约 B（如交易所）
  │
  │─── approve(B, 1000) ────▶│  授权 1000 代币给 B
  │                          │
  │                     授权额度：1000
  │                          │
  │◀─── transferFrom(A) ─────│  B 从 A 的账户转出代币
  │     (转走代币)            │
```

**为什么需要授权转账？**

在 DeFi 场景中（如交易所、借贷协议）：
1. 用户授权合约可以提取自己的代币
2. 合约才能自动执行交易（如 swap、流动性挖矿等）

### 1.4 发行代币的流程

```
1. 编写一个 ERC20 合约（遵循标准）
      │
      ▼
2. 部署到区块链上（相当于"印刷"）
      │
      ▼
3. 初始供应量发给部署者
      │
      ▼
4. 授权给交易所 → 可以流通交易了
```

> **总结**：ERC20 就是代币的"流通规则"，遵守这个规则 → 代币就能被钱包识别、被交易所上架、在 DeFi 中使用。

### 2.5 编写智能合约

在 `contracts/` 目录下创建 `.sol` 文件：

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MyToken is ERC20 {
    constructor() ERC20("MyToken", "MTK") {
        _mint(msg.sender, 1000000 * 10 ** decimals());
    }
}
```

### 2.6 编写测试用例

#### 测试框架

- **Hardhat** - 提供测试环境
- **Chai** - 断言库（`expect(...).to.equal(...)`）
- **Ethers.js** - 以太坊交互库

#### 测试文件结构

```javascript
const { expect } = require("chai");
const { ethers } = require("hardhat");

describe("合约名", function () {
  let contract;
  let owner, addr1, addr2;

  beforeEach(async function () {
    [owner, addr1, addr2] = await ethers.getSigners();
    const Factory = await ethers.getContractFactory("合约名");
    contract = await Factory.deploy(...参数);
    await contract.deployed();
  });

  describe("功能模块", function () {
    it("测试描述", async function () {
      // 测试逻辑
      expect(结果).to.equal(预期);
    });
  });
});
```

#### 测试规律

| 类型 | 说明 |
|------|------|
| **正向** | 正常情况应该成功 |
| **负向** | 异常情况应该失败 |

```javascript
// 正向测试
it("应该允许转账代币", async function () {
  await contract.transfer(addr1.address, 100);
  expect(await contract.balanceOf(addr1.address)).to.equal(100);
});

// 负向测试
it("应该拒绝余额不足的转账", async function () {
  await expect(
    contract.transfer(addr1.address, 99999999)
  ).to.be.revertedWith("ERC20: insufficient balance");
});
```

### 2.7 编写部署脚本

在 `scripts/` 目录下创建 `deploy.js`：

```javascript
const hre = require("hardhat");
const { ethers } = require("hardhat");

async function main() {
  // 1. 获取测试账户
  const [deployer, addr1, addr2] = await ethers.getSigners();

  // 2. 编译合约
  await hre.run("compile");

  // 3. 获取合约工厂并部署
  const ContractFactory = await ethers.getContractFactory("合约名");
  const contract = await ContractFactory.deploy(...参数);
  await contract.deployed();

  console.log("合约地址:", contract.address);
}

main()
  .then(() => process.exit(0))
  .catch((error) => {
    console.error(error);
    process.exit(1);
  });
```

## 3. ERC20 合约代码详解

### 3.1 ERC20Basic.sol - 手写 ERC20 实现

#### 核心状态变量

```solidity
string public name;           // 代币名称
string public symbol;         // 代币符号
uint8 public decimals;        // 小数位数（通常 18）
uint256 public totalSupply;   // 总供应量

mapping(address => uint256) public balanceOf;  // 地址 → 余额
mapping(address => mapping(address => uint256)) public allowance;  // 授权映射
```

#### 核心函数

| 函数 | 作用 |
|------|------|
| `transfer()` | 用户自己转账 |
| `approve()` | 授权他人 |
| `transferFrom()` | 被授权者转账 |
| `increaseAllowance()` | 增加授权（安全） |
| `decreaseAllowance()` | 减少授权 |

### 3.2 MyERC20Token.sol - 使用 OpenZeppelin

#### 引入依赖

```solidity
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import "@openzeppelin/contracts/access/Ownable.sol";
```

#### 继承关系

```
┌─────────────────┐
│  Ownable        │  ← 权限管理
└────────┬────────┘
         │
┌────────▼────────┐
│  ERC20          │  ← 代币标准
└────────┬────────┘
         │
┌────────▼────────┐
│  MyERC20Token  │  ← 你的代币
└─────────────────┘
```

#### 自定义功能

| 函数 | 说明 |
|------|------|
| `mint()` | 铸造新代币（仅 owner） |
| `burn()` | 销毁自己的代币 |
| `burnFrom()` | 强制销毁（仅 owner） |
| `finishMinting()` | 结束铸造（仅 owner） |

### 3.3 _mint() 和 _burn() 的来源

这两个方法定义在 **OpenZeppelin 的 ERC20.sol** 中，通过继承引入：

```solidity
function _mint(address account, uint256 amount) internal {
    require(account != address(0), "ERC20: mint to zero address");
    _totalSupply += amount;
    _balances[account] += amount;
    emit Transfer(address(0), account, amount);
}
```

| 函数 | 类型 | 说明 |
|------|------|------|
| `mint()` | public | 公开接口，内部调用 _mint |
| `_mint()` | internal | 内部实现 |

### 3.4 为什么不使用 super._mint()？

`super.函数()` 用于调用**父类中被重写**的函数。

```solidity
// 如果没有重写，直接调用即可
_mint(to, amount);        // ✅ 推荐

// 如果重写了，需要调用父类版本
super._mint(to, amount);  // 调用父类
```

### 3.5 手写 vs OpenZeppelin 对比

| 特性 | ERC20Basic（手写） | MyERC20Token（OpenZeppelin） |
|------|-------------------|------------------------------|
| 安全性 | 需要自己处理 | 经过审计，更安全 |
| 代码量 | 较多 | 较少 |
| 继承 | 无 | `ERC20`, `Ownable` |
| 铸造功能 | 无 | 有 `mint()` |
| 销毁功能 | 无 | 有 `burn()` |
| 权限控制 | 无 | 有 `onlyOwner` |

**核心优势**：使用 OpenZeppelin = 用经过审计的安全代码。
