02 整数溢出

什么是整数溢出？

整数溢出是指在进行算术运算时，结果超出了数据类型的表示范围，导致数值回绕的错误

溢出类型

  • 上溢（Overflow）：当值超过最大值时，回绕到最小值
  • 例如：uint8 max = 255 → 255 + 1 = 0（发生上溢）
  • 下溢（Underflow）：当值低于最小值时，回绕到最大值
  • 例如：uint8 min = 0 → 0 - 1 = 255（发生下溢）

02 整数溢出

Solidity 版本差异

0.8.0 之前

  • 整数溢出不会抛出异常
  • 溢出会被静默忽略
  • 需要手动使用 SafeMath 库

0.8.0 及之后

  • 默认启用溢出检查
  • 溢出会自动 revert 交易
  • 可使用 unchecked 块禁用检查

02 整数溢出

什么是 SafeMath？

SafeMath 是 OpenZeppelin 提供的数学运算库，通过显式的溢出检查防止整数溢出问题 

// SafeMath 库核心实现
library SafeMath {
    // 安全的加法运算
    function add(uint256 a, uint256 b) internal pure returns (uint256) {
        uint256 c = a + b;
        require(c >= a, "SafeMath: addition overflow");  // 检查是否溢出
        return c;
    }

    // 安全的减法运算
    function sub(uint256 a, uint256 b) internal pure returns (uint256) {
        require(b <= a, "SafeMath: subtraction overflow");  // 检查是否下溢
        return a - b;
    }

    // 安全的乘法运算
    function mul(uint256 a, uint256 b) internal pure returns (uint256) {
        if (a == 0) return 0;
        uint256 c = a * b;
        require(c / a == b, "SafeMath: multiplication overflow");  // 检查是否溢出
        return c;
    }
}

02 整数溢出

漏洞代码详解

旧版本 Solidity (< 0.8) 批量转账存在整数溢出漏洞

// SPDX-License-Identifier: MIT
pragma solidity ^0.7.0;

contract Token {
    // 代币余额映射
    mapping(address => uint256) public balances;

    // ⚠️ 批量转账存在整数溢出漏洞
    function batchTransfer(address[] memory receivers, uint256 value) public {
        // 计算总金额 = 单价 × 人数
        uint256 totalAmount = value * receivers.length;  // 可能溢出！
        // 溢出情况下 totalAmount 变成 0

        require(balances[msg.sender] >= totalAmount, "Insufficient balance");
        // 溢出后 0 <= 任何余额，检查永远通过！

        balances[msg.sender] -= totalAmount;

        for (uint256 i = 0; i < receivers.length; i++) {
            balances[receivers[i]] += value;
        }
    }
}

02 整数溢出

攻击流程解释

步骤 正常情况 攻击情况
1. 输入 value=100, 2人 value=2^255, 2人
2. totalAmount 200 2^256 → 溢出→0
3. 余额检查 余额≥200 余额≥0 → 永远通过
4. 扣除发送者 余额-200 余额-0 = 不变
5. 给接收者 每人+100 每人+2^255

关键：2^255 × 2 = 2^256，超出 uint256 最大值，溢出后 totalAmount = 0

02 整数溢出

unchecked 块

unchecked 块是 Solidity 0.8+ 提供的功能，允许在特定代码块中禁用溢出检查 

// 使用 unchecked 块的示例
function uncheckedExample(uint256 a, uint256 b) public pure returns (uint256) {
    unchecked {
        return a + b;  // 不检查溢出
    }
}

⚠️ 注意： 仅在确定不会溢出时使用 unchecked 块

03 访问控制

什么是访问控制？

访问控制是指限制谁可以调用特定函数的机制。未授权的访问可能导致资金被盗、状态被恶意修改

常见访问控制问题

  • 缺少权限检查
  • 权限检查逻辑错误
  • 使用 tx.origin 而非 msg.sender
  • 权限可以被任意设置

03 访问控制

msg.sender vs tx.origin

变量 含义
msg.sender 当前调用的直接调用者（可以是用户或合约）
tx.origin 交易的原始发起者（最开始的调用者，永远不变）

调用链示例

用户A (0xAAA) → 恶意合约B (0xBBB) → 目标合约C (0xCCC)

在合约C中：
tx.origin = 0xAAA (用户A，永远不变)
msg.sender = 0xBBB (恶意合约，当前直接调用者)

03 访问控制

为什么 tx.origin 不安全？

不是因为 tx.origin 会变，恰恰相反——它太"死板"了，只认最开始的地址 

// ⚠️ 不安全：使用 tx.origin
function withdraw() public {
    require(tx.origin == owner, "Not owner");
    payable(tx.origin).transfer(address(this).balance);
}

// ✅ 安全：使用 msg.sender
function withdraw() public {
    require(msg.sender == owner, "Not owner");
    payable(msg.sender).transfer(address(this).balance);
}

攻击：用户A (owner) → 恶意合约B → 目标合约C
tx.origin = 用户A = owner ✅ 通过！但 msg.sender = 恶意合约B ← 真正的调用者！

03 访问控制

什么是 Ownable？

Ownable 是 OpenZeppelin 提供的访问控制合约，通过 onlyOwner 修饰符限制特定函数只能由合约所有者调用 

contract Ownable {
    address public owner;

    // 构造函数：部署者成为所有者
    constructor() {
        owner = msg.sender;
    }

    // onlyOwner 修饰符：检查调用者是否为所有者
    modifier onlyOwner() {
        require(msg.sender == owner, "Ownable: caller is not the owner");
        _;  // 执行函数体
    }

    // 转移所有权
    function transferOwnership(address newOwner) public onlyOwner {
        owner = newOwner;
    }
}

03 访问控制

什么是 AccessControl？

AccessControl 是 OpenZeppelin 提供的更细粒度的访问控制机制，支持多角色管理 

// 使用 AccessControl 的示例
contract MyContract is AccessControl {
    // 定义角色
    bytes32 public constant ADMIN_ROLE = keccak256("ADMIN_ROLE");
    bytes32 public constant MINTER_ROLE = keccak256("MINTER_ROLE");

    // 管理员函数
    function adminFunction() public onlyRole(ADMIN_ROLE) {}

    // 铸币函数
    function mint() public onlyRole(MINTER_ROLE) {}
}

💡 最小权限原则：每个角色只应该被授予完成其任务所必需的最小权限

04 重入攻击

什么是重入攻击？

重入攻击发生在状态更新之前执行外部调用时。攻击者利用回调函数再次进入原函数，导致状态未及时更新

涉及的角色

  • 角色1：攻击者（人）- 部署恶意合约
  • 角色2：Attacker 合约（恶意合约）
  • 角色3：VulnerableBank（受害合约）

04 重入攻击

攻击流程详解

// 有重入漏洞的提现函数
function withdraw() public {
    uint256 amount = balances[msg.sender];
    require(amount > 0);

    (bool success, ) = msg.sender.call{value: amount}("");
    // ↑ 这里会"暂停" withdraw()，去执行 Attacker 的 receive()

    // ← receive() 执行完后才会回到这里
    balances[msg.sender] = 0;  // ← 不知道什么时候才能执行到
}

时间线：withdraw() → call() → receive() → withdraw() → call() → receive() → ...
形成递归，直到余额=0 或 Gas耗尽

04 重入攻击

Checks-Effects-Interactions 模式

安全编码三步曲

  1. Checks（检查）：验证所有前置条件
  2. Effects（效果）：更新所有状态变量
  3. Interactions（交互）：执行外部调用

为什么有效？因为状态在外部调用之前已经更新，攻击者重入时余额已经是 0 了 

// ✅ 安全的写法：先更新状态，再发送ETH
function withdraw() public {
    require(balances[msg.sender] > 0, "No balance");  // 1. 检查
    uint256 amount = balances[msg.sender];
    balances[msg.sender] = 0;  // 2. 先更新状态！
    payable(msg.sender).transfer(amount);  // 3. 再发送ETH
}

04 重入攻击

什么是 ReentrancyGuard？

ReentrancyGuard 是 OpenZeppelin 提供的重入保护机制，通过"互斥锁"防止函数被递归调用 

// ReentrancyGuard 核心实现
contract ReentrancyGuard {
    uint256 private locked = 1;  // 锁状态：1=未锁定，2=已锁定

    // 非重入修饰符
    modifier nonReentrant() {
        require(locked == 1, "ReentrancyGuard: reentrant call");
        locked = 2;  // 上锁
        _;
        locked = 1;  // 解锁
    }
}

💡 使用方式：在函数上添加 nonReentrant 修饰符

04 重入攻击

什么是 Pull Payment 模式？

让用户主动提取资金，而不是由合约主动发送 

// Pull Payment 模式示例
contract PullPayment {
    // 待提取金额映射
    mapping(address => uint256) public payments;

    // 记录待提取的金额（合约主动调用）
    function _asyncTransfer(address to, uint256 amount) internal {
        payments[to] += amount;
    }

    // 用户主动提取（避免重入）
    function withdraw() public {
        uint256 amount = payments[msg.sender];
        require(amount > 0);
        payments[msg.sender] = 0;  // 先清零
        payable(msg.sender).transfer(amount);  // 后转账
    }
}

08 Q&A

常见问题解答

Q1: value * receivers.length 为什么会有溢出风险？

A: 当 value 非常大时，乘以 receivers.length 可能超过 uint256 最大值。例如：value = 2^255，receivers.length = 2，2^255 × 2 = 2^256 > uint256.max，溢出结果变成 0

Q2: receivers.length 是什么？

A: receivers 是传入的地址数组参数，receivers.length 是数组长度

08 Q&A

常见问题解答（续）

Q3: msg.sender 中的 "sender" 是什么意思？

A: "sender" 是"发送者"的意思，指"发送这笔交易的人"。类比：寄信人 = msg.sender，邮局 = 以太坊网络，收信人 = 合约地址

Q4: tx.origin == owner 判断是永久成立的吗？

A: 是的，tx.origin 永远指向交易的最初发起者，永远不变。但这恰恰是问题！攻击者利用用户是 owner 的身份，通过恶意合约间接调用目标合约，目标检查 tx.origin 发现是 owner 就通过了

08 Q&A

常见问题解答（续）

Q5: 重入攻击中时间间隔很短，攻击者怎么实现多次调用？

A: 不是"快速多次"，而是"递归调用"！

  1. withdraw() 执行到 call()
  2. call() 触发 receive()
  3. receive() 中又调用 withdraw()
  4. 形成递归循环

直到：银行余额=0 或 Gas耗尽

Q6: Chainlink 是中心化的数据中心吗？

A: 不是。Chainlink 是去中心化的节点网络：全球分布的多个独立节点，每个节点独立从多个数据源获取数据，数据聚合后才返回
