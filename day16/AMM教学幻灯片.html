Day 16

AMM 与 x·y=k 恒定乘积模型

自动做市商 | 流动性池 | 代币交换

基于本项目合约提炼的知识点

1

什么是 AMM？

自动做市商（AMM）：用数学公式和池子里的储备量
自动给出买卖价格，无需传统订单簿和做市商报价

类比：像一个自动售货机——你投进去的是代币A，机器按内部规则吐出代币B。规则写死在合约里，不需要有人在场报价。

  • 传统交易所：订单簿 + 撮合。有人挂卖单、有人挂买单，交易所撮合成交
  • AMM：没有挂单。有一个流动性池，里面存着两种代币
2

x·y=k 恒定乘积模型

池子里两种代币的数量（x 和 y）的乘积始终等于常数 k

amountOut = reserveOut × amountIn / (reserveIn + amountIn)

公式推导

  • 设池子里有 x 个 A、y 个 B
  • 用户投入 Δx 个 A，要拿走 Δy 个 B
  • 交换后：(x+Δx)(y−Δy) = k
  • 解得：Δy = y·Δx / (x+Δx)

类比

像跷跷板：一边加重（投入 A），另一边就必须减少（取出 B），且"重量×距离"的乘积保持不变

3

x·y=k 核心代码

// 仅演示 x·y=k 的输入输出计算（view，不改变状态） function getAmountOut( uint256 reserveIn, uint256 reserveOut, uint256 amountIn ) public pure returns (uint256 amountOut) { // 恒定乘积：(reserveIn + amountIn) * (reserveOut - amountOut) = reserveIn * reserveOut // 解得：amountOut = reserveOut * amountIn / (reserveIn + amountIn) require(reserveIn > 0 && reserveOut > 0, "no liquidity"); amountOut = (reserveOut * amountIn) / (reserveIn + amountIn); return amountOut; }

最佳实践：先乘后除，避免先除再乘导致精度丢失

4

添加流动性（Add Liquidity）

用户把两种代币按约定比例存入 AMM 池子，获得流动性份额（LP 代币）

首次添加

  • 池子为空，可任意定初始比例
  • LP 数量 = √(amountA×amountB) − 最小流动性

后续添加

  • 必须按当前池子比例
  • amountA/amountB = reserveA/reserveB
  • LP = (amountA × totalSupply) / reserveA
LP = (amountA × totalSupply) / reserveA
5

移除流动性（Remove Liquidity）

用户销毁一定数量的 LP 代币，按当前池子比例取回两种代币

amountA = liquidity × reserveA / totalSupply
amountB = liquidity × reserveB / totalSupply
function removeLiquidity(uint256 _liquidity) external returns (uint256 amountA, uint256 amountB) { // 按份额比例计算可取回的数量 amountA = (_liquidity * reserveA) / totalSupply; amountB = (_liquidity * reserveB) / totalSupply; require(amountA > 0 && amountB > 0, "Insufficient amounts"); // CEI：先更新状态，再转币 _burn(msg.sender, _liquidity); reserveA -= amountA; reserveB -= amountB; IERC20(tokenA).transfer(msg.sender, amountA); IERC20(tokenB).transfer(msg.sender, amountB); return (amountA, amountB); }

关键点

  • LP 代表池子份额
  • 按比例取回，不改变剩余储备比例
  • 使用 CEI 模式避免重入
6

LP 代币（流动性凭证）

代表你在该流动性池里所占的份额

添加流动性

合约 _mint(msg.sender, liquidity) 铸造 LP 给用户

移除流动性

合约 _burn(msg.sender, liquidity) 销毁用户的 LP

赎回比例

按 LP 占总供应量的比例取回双币

最小流动性

首次铸造减掉 MINIMUM_LIQUIDITY 锁到 address(0)，避免除零

// 最简 LP 记账 uint256 public totalSupply; mapping(address => uint256) public balanceOf; function _mint(address to, uint256 amount) internal { totalSupply += amount; balanceOf[to] += amount; emit Transfer(address(0), to, amount); }

代币交换（Swap）

用户把一种代币转入池子，合约按 x·y=k 算出另一种代币的数量并转给用户

// 实际交换：转入 tokenIn，更新储备，转出 tokenOut function swap(address _tokenIn, uint256 _amountIn) external returns (uint256 amountOut) { (uint256 rIn, uint256 rOut) = _tokenIn == tokenA ? (reserveA, reserveB) : (reserveB, reserveA); amountOut = (rOut * _amountIn) / (rIn + _amountIn); require(amountOut > 0, "Bad amount"); // 先转进，再更新储备，最后转出 IERC20(_tokenIn).transferFrom(msg.sender, address(this), _amountIn); if (_tokenIn == tokenA) { reserveA += _amountIn; reserveB -= amountOut; IERC20(tokenB).transfer(msg.sender, amountOut); } else { reserveB += _amountIn; reserveA -= amountOut; IERC20(tokenA).transfer(msg.sender, amountOut); } return amountOut; }
8

滑点（Slippage）与最小输出

从发起交易到上链之间，池子可能被其他交易改变，导致实际得到的 amountOut 小于预期

什么是滑点？

你调用 getAmountOut 时得到 100 个 B；在交易之前，有人先换走了大量 B，池子比例变了，你实际可能只换到 95 个 B。这 5% 的偏差就是滑点。

三明治攻击

有人监控到你的交易还没执行，先用大单买 B → 你的交易在更贵价格执行 → 他再卖 B 赚差价。你的单子被夹在中间。

// 带滑点保护的 swap function swap( address _tokenIn, uint256 _amountIn, uint256 minAmountOut // 最少要拿到多少 ) external returns (uint256 amountOut) { amountOut = (reserveOut * _amountIn) / (reserveIn + _amountIn); require(amountOut >= minAmountOut, "Slippage too high"); // 再执行转入、更新储备、转出... }
9

ERC20 的 Approve 与 TransferFrom

AMM 需要把用户的代币从用户地址转到池子

用户操作

  • approve(spender, amount)：允许 spender 代表你转账
  • 用户必须先对 AMM 合约 approve
  • 添加流动性：approve tokenA 和 tokenB
  • 交换：approve tokenIn

AMM 内部操作

  • transferFrom(from, to, amount)
  • 从 from 转 amount 到 to
  • 前提：allowance[from][spender] ≥ amount
  • 调用：IERC20(token).transferFrom(msg.sender, address(this), amount)
类比：你把钱包交给朋友（AMM）代付，需要先说"最多可以花 100 块"（approve）
10

整数运算与最小流动性

除零问题

  • totalSupply 或 reserve 为 0 时，除法会 revert
  • Uniswap 用最小流动性解决
  • 首次铸造时把少量 LP（如 10^3）铸给 address(0)
  • 使 totalSupply 永远 ≥ MINIMUM_LIQUIDITY

舍入问题

  • Solidity 只有整数运算，向下取整
  • 实际 amountOut 略少
  • k 略增，对 LP 有利
  • 设计上可接受
uint256 public constant MINIMUM_LIQUIDITY = 10**3; // 首次添加：扣掉最小流动性并锁到 0 地址 if (reserveA == 0 && reserveB == 0) { liquidity = sqrt(_amountA * _amountB) - MINIMUM_LIQUIDITY; _mint(address(0), MINIMUM_LIQUIDITY); }
11

安全与校验

CEI 模式

Checks-Effects-Interactions：先做所有检查和状态更新，最后再做外部调用。防止重入攻击。

参数校验

token 地址非零、两 token 不同、amount > 0、reserve 充足、amountOut < reserveOut 等

事件完整性

LiquidityAdded、LiquidityRemoved、Swap 等要发全参数，便于链下索引和审计

最佳实践

构造函数中：require(_tokenA != address(0) && _tokenB != address(0) && _tokenA != _tokenB)

小镇的换货站：故事开始

一、老集市：订单簿与做市商

小镇以前只有老集市。想卖苹果的人在一块大木板上写「我卖 10 斤苹果，5 元一斤」；想买的人写「我买 5 斤苹果，4 元一斤」。管理员把买和卖一对一对配起来，配上了就成交。

  • 那块木板就是订单簿
  • 写上去的每一句就是挂单
  • 有些摊主主动写「我 4.5 元收、5.5 元卖」，就是在做市
  • 这些摊主就是做市商
13

小镇的换货站

二、新式换货站：AMM 与流动性池

后来镇里开了新式换货站。门口只有两个大桶：一个装苹果（ETH）、一个装橙子（USDT），桶旁边贴着一张纸，写着一道规则：「两桶里的数量乘起来，永远等于同一个数 k。」

  • 没有人再在木板上写买价卖价了
  • 想换的人把一种水果倒进对应桶里，按规则从另一个桶里取出水果
  • 这个换货站就是自动做市商 AMM
  • 两个桶里已经装着的数量就是储备量
  • 桶越满，流动性越好
14

小镇的换货站

四、添加流动性与 LP 代币

开张第一天，小明和小红各往两个桶里倒了一批苹果和橙子。站员按规则算了一下，给小明和小红发了小票，上面写着「凭此票可按比例从桶里取回苹果和橙子」。

  • 这些小票就是LP 代币
  • 代表你在流动性池里占的份额
  • 以后谁再来补货或有人来换货，桶里的数量和总小票数会变
  • 你手里的票数不变，但你占总票数的比例决定了你以后能取回多少
  • 往桶里补货的过程，就是添加流动性
15

小镇的换货站

五、交换：公式与滑点

小刚想用苹果换橙子。站员算出来：「能拿 8 斤橙子。」那条规则就是：

amountOut = 橙子桶数量 × 苹果数量 ÷ (苹果桶数量 + 苹果数量)

滑点

小刚发现实际只拿到了 7.5 斤，比问的 8 斤少——这中间的差距就是滑点。

三明治攻击

有人先冲进去大量用苹果换橙子 → 小刚的交易在更差汇率执行 → 这人再把橙子换回苹果赚差价。

后来换货站规定：可以说「我至少要拿到 X 斤，否则作废」——这个"至少 X 斤"就是minAmountOut

16

小镇的换货站

六、移除流动性：可能多，也可能少

小明不想再参与这个站了，就拿出手里的小票到换货站说「我要赎回」。站员按小票占总票数的比例，从两个桶里分别舀出对应数量的水果还给他，同时把小票撕掉。

  • 这一过程就是移除流动性
  • 小票被撕掉相当于销毁 LP 代币
  • 取回量可能比当初投入的少，也可能多
  • 少掉的水果被来换货的人换走了——这就是无常损失
17

小镇的换货站

七、风险与补偿：手续费能否覆盖不确定

做 AMM 是有风险的：取回时可能比当初投入的多，也可能少。每一笔换货，站里都会扣一点手续费（例如 0.3%），这部分钱会按比例分给所有持小票的人。

  • 补偿 = 你作为 LP 分到的那部分交易手续费
  • 手续费能不能覆盖无常损失，要看：
  • 手续费率多高、换货量多大、两种水果的相对价格波动多大
  • 有时能覆盖还有赚，有时覆盖不了

概念速查

订单簿

交易所的"登记册"，记录买卖单，撮合成交。AMM 没有订单簿。

做市商

主动报买卖价并愿意同时接买和卖的一方。AMM 用公式做市。

流动性

池子里代币的数量。流动性好 = 稍微买/卖价格不会乱跳。

储备量

AMM 池子里当前实际存放的每种代币的数量。

LP 代币

流动性凭证，代表你在池子里占的份额，可按比例赎回双币。

滑点

预期可得数量 − 实际得到数量，用 minAmountOut 保护。
