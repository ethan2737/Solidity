闪电贷

Flash Loan

回调机制强化理解版

Day 18 - Solidity 学习

核心结论

先记住这5句话

  • 借款人必须调用池子的 flashLoan 函数才能借钱
  • 一旦调用 flashLoan，控制权就交给池子合约
  • onFlashLoan 是池子在执行过程中"回调"借款人的函数
  • 套利等操作必须在 onFlashLoan 回调中完成
  • 必须在回调结束前归还本金+手续费，否则整个交易回滚

什么是闪电贷？

闪电贷是 DeFi 领域的一种创新金融工具，
允许用户在同一笔区块链交易中
无需抵押任何资产即可借入大量资金

关键特性

  • 无抵押借款 — 传统借贷需要超额抵押，闪电贷无需抵押
  • 单笔交易完成 — 借款、还款必须在同一个区块的一笔交易中完成
  • 原子性操作 — 如果还款失败，整个交易回滚，就像什么都没发生过

术语解释

术语解释
Flash Loan闪电贷，一种无抵押的即时贷款机制
Liquidity Pool流动性池，存储可供借贷的资金池
Borrower借款人，通过闪电贷借入资金的主体
Fee手续费，闪电贷借款人需要支付的费用
Callback回调函数，借款成功后被调用的函数

角色与职责

池子合约

  • 提供 flashLoan 函数
  • 借钱能力
  • 验证还款

借款人合约

  • 调用 flashLoan 发起借款
  • 实现 onFlashLoan 回调函数
  • 执行套利操作

最容易搞错的点

❌ 错误理解

  • 我要实现 flashLoan 才能借钱
  • 回调函数是我主动调用的
  • 池子在"监控"我

✅ 正确理解

  • 我调用 flashLoan（发起借款）
  • 池子执行整个流程
  • 池子在过程中调用我的 onFlashLoan
  • 我的逻辑是在池子的执行流程中被执行

执行控制权

关键理解

❌ 不是实现 onFlashLoan 接口的时候

✅ 而是调用 flashLoan 的那一刻

你 → 池子 → 你（回调） → 池子

你只是"被调用的一段逻辑"，不是主导者

回调函数本质

重新定义

回调函数 = 池子合约在执行过程中，
强制调用借款人执行的一段逻辑入口

完整调用链

最重要

你（借款人）调用 flashLoan ↓ 池子执行 ↓ 转钱给你 ↓ 调用 onFlashLoan（你） ↓ 你执行套利 ↓ 返回池子 ↓ 池子检查还款 ↓ 交易结束 或 回滚

真实本质

升维理解

❌ 直觉理解

借钱 → 操作 → 还钱

✅ 本质理解

在池子的一次函数执行过程中，
插入一段你的逻辑

为什么会有闪电贷？

传统借贷的问题

  • 要借入价值 80 美元的资产，需要存入价值 100 美元的加密货币作为抵押
  • 如果违约，损失全部抵押品
  • 问题：资金效率低，大量资产被锁定

闪电贷的创新

  • 资金效率最大化 — 无需抵押，资金零成本借出
  • 套利机会 — 利用不同 DEX 之间的价格差异进行无风险套利
  • 降低门槛 — 任何人都可以参与复杂的金融操作
  • 创新基础 — 催生了闪电贷攻击、即时杠杆等新玩法

应用场景

场景描述
套利利用不同交易所的价格差获利
抵押品交换将一种资产转换为另一种资产再还款
再融资在同一交易中偿还旧贷款并获取新贷款
价格操纵通过大额交易影响价格（常用于攻击）

为什么套利必须在回调中完成？

因为只有在回调执行时：

  • 钱已经到账 ✅
  • 交易还没结束 ✅
  • 可以自由操作 ✅

原子性

最关键机制

借钱 + 使用 + 还钱 = 必须在同一笔交易中完成

如果失败：

  • 没还钱 ❌
  • 少还钱 ❌
  • 套利失败 ❌

💥 整个交易回滚（什么都没发生）

核心合约代码

SimpleFlashLoan.sol

contract SimpleFlashLoan {
    using SafeERC20 for IERC20;

    uint256 public constant FLASH_LOAN_FEE_BPS = 9;  // 0.09%

    function flashLoan(address token, uint256 amount, bytes calldata data) external {
        uint256 balanceBefore = tokenContract.balanceOf(address(this));
        require(balanceBefore >= amount, "Insufficient liquidity");

        uint256 fee = (amount * FLASH_LOAN_FEE_BPS) / 10000;

        // 借出资金
        tokenContract.safeTransfer(msg.sender, amount);

        // 调用回调函数
        IFlashLoanReceiver(msg.sender).onFlashLoan(token, amount, fee, data);

        // 验证还款
        uint256 balanceAfter = tokenContract.balanceOf(address(this));
        require(balanceAfter >= balanceBefore + fee, "Flash loan not repaid");
    }
}

关键步骤详解

步骤 1-3

  • 步骤1：记录初始余额 - 在借出前记录代币余额，验证还款的依据
  • 步骤2：验证流动性 - 确保池子有足够资金，防止超额借出
  • 步骤3：计算手续费 - 借款金额 × 费率（0.09%）
// 手续费计算示例
uint256 fee = (amount * 9) / 10000;  // 0.09%

关键步骤详解

步骤 4-6

  • 步骤4：转移资金 - 使用 SafeERC20 安全转账
  • 步骤5：回调执行 - 调用借款人的 onFlashLoan 函数
  • 步骤6：验证还款 - 检查余额是否 >= 初始余额 + 手续费

核心安全机制：如果不足，整个交易回滚

正面效果

  • 无本套利 — 零资金成本进行套利操作，利润归借款人
  • 资金效率 — 不需要锁定大量抵押资产
  • 创新机会 — 催生了 DeFi 领域的各种创新应用
  • 流动性 — 提高市场流动性，促进价格发现

负面影响（风险）

  • 闪电贷攻击 — 攻击者利用漏洞进行套利，导致协议损失
  • 价格操纵 — 大额交易人为改变价格
  • 智能合约漏洞 — 如果回调合约有漏洞，可能导致资金损失
  • gas 限制 — 复杂操作可能超出区块 gas 限制

总结

核心要点

  • 闪电贷是什么？ — 无抵押、单笔交易完成、原子性的借贷机制
  • 为什么存在？ — 提高资金效率，创造套利机会，推动 DeFi 创新
  • 怎么做？ — 实现 IFlashLoanReceiver 接口，在回调中执行操作并归还资金
  • 效果如何？ — 可用于套利、再融资等，也存在安全风险

一句话总结

面试级

借款人调用 flashLoan 发起借款后，控制权进入池子合约，
池子在执行过程中回调借款人的 onFlashLoan，
借款人需要在该回调中完成操作并归还资金，
最终由池子验证还款，否则整个交易回滚。

最终总结

闪电贷不是"借钱"，而是：

在别人控制的执行流程里，
短暂使用资金完成你的逻辑
